# Phase 3: Research & Validation

## เป้าหมาย
ทดสอบสมมติฐานสำคัญก่อนสร้าง product จริง เพื่อไม่ให้เสียเวลาและเงินไปกับสิ่งที่ไม่มีคนต้องการ

---

## Hypothesis-First Thinking

ทุกอย่างที่เราเชื่อเกี่ยวกับ product คือ "สมมติฐาน" จนกว่าจะ validate ได้

### สมมติฐานแบ่งเป็น 3 ระดับ

| ระดับ | ประเภท | ตัวอย่าง |
|-------|--------|---------|
| **Critical** | ถ้าผิดธุรกิจล้มเลย | "คนยินดีจ่าย X บาทสำหรับ solution นี้" |
| **Important** | ถ้าผิดต้องปรับแผนใหญ่ | "ช่องทาง acquire ลูกค้าคือ social media" |
| **Nice-to-know** | ถ้าผิดปรับ feature ได้ | "ผู้ใช้ชอบ dark mode" |

**กฎ:** Validate สมมติฐาน Critical ก่อนเสมอ

---

## Hypothesis Template

```
สมมติฐาน: [ระบุสิ่งที่เชื่อ]
ระดับความสำคัญ: Critical / Important / Nice-to-know
วิธีทดสอบ: [วิธีที่เร็วและถูกที่สุด]
เกณฑ์สำเร็จ: [ตัวเลขหรือสัญญาณที่บอกว่า validate แล้ว]
เวลาที่ใช้: [กี่วัน/สัปดาห์]
ผลที่ได้: [บันทึกหลังทดสอบ]
```

**ตัวอย่าง:**
```
สมมติฐาน: Freelance Designer ยินดีจ่าย 299 บาท/เดือน สำหรับ proposal generator
ระดับ: Critical
วิธีทดสอบ: Landing page + waitlist + pre-order ในราคานั้น
เกณฑ์สำเร็จ: 50 คน sign up ใน 2 สัปดาห์
เวลาที่ใช้: 2 สัปดาห์
```

## User Interview Guide

### การเตรียม Interview
- สัมภาษณ์ **5-10 คน** จาก target customer segment
- ใช้เวลา **20-30 นาที** ต่อคน
- บันทึกด้วยความยินยอม

### คำถามที่ควรใช้ (Mom Test Style)

**อย่าถามเกี่ยวกับ product ของเราโดยตรง — ถามเกี่ยวกับชีวิตและปัญหาของเขา**

```
ส่วนที่ 1: เปิดบทสนทนา
- "ช่วยเล่าให้ฟังหน่อยได้ไหมว่า [งาน/ปัญหาที่สนใจ] เป็นยังไงในชีวิตประจำวันของคุณ?"

ส่วนที่ 2: ค้นหาปัญหา
- "ส่วนที่ยากที่สุดของ [งานนั้น] คืออะไร?"
- "ครั้งสุดท้ายที่เจอปัญหานี้คือเมื่อไหร่? เกิดอะไรขึ้น?"
- "คุณแก้ปัญหานี้ยังไงอยู่ตอนนี้?"

ส่วนที่ 3: เข้าใจ motivation
- "ทำไมถึงสำคัญกับคุณ?"
- "ถ้าปัญหานี้หายไปวันนี้เลย ชีวิตคุณจะต่างออกไปยังไง?"

ส่วนที่ 4: ตรวจสอบความพร้อมจ่าย
- "ตอนนี้คุณใช้เงินกับเรื่องนี้อยู่บ้างไหม? เท่าไหร่?"
- "ถ้ามีวิธีแก้ปัญหานี้ที่ดีกว่า คุณจะพิจารณาจ่ายเงินสำหรับมันไหม?"
```

### สิ่งที่ห้ามทำใน Interview
- ❌ อย่าถามว่า "คุณจะใช้ product ของฉันไหม?" (คนจะตอบ yes เพื่อให้ใจ)
- ❌ อย่า pitch product ของตัวเอง
- ❌ อย่าถามเกี่ยวกับอนาคต — ถามเกี่ยวกับอดีตและปัจจุบัน

## MVP Definition

### MVP คืออะไร
MVP = Minimum Viable Product — version เล็กที่สุดที่ validate สมมติฐาน Critical ได้

**MVP ไม่ใช่** product ที่ครบทุก feature แต่ไม่ polish
**MVP คือ** วิธีทดสอบที่เร็วและถูกที่สุด

### MVP Types (เรียงจากเร็วไปช้า)

| ประเภท | คำอธิบาย | เวลา |
|--------|----------|------|
| **Concierge MVP** | ทำ manual ทั้งหมดโดยไม่มี software | 1-3 วัน |
| **Wizard of Oz** | หน้าบ้านดูเหมือน automated แต่จริงๆ ทำมือ | 3-7 วัน |
| **Landing Page** | หน้าเว็บ + waitlist เพื่อวัด interest | 1-3 วัน |
| **Smoke Test** | Ad campaign ไปหน้า pre-order | 1-2 สัปดาห์ |
| **Prototype** | Clickable mockup ทดสอบ UX | 1-2 สัปดาห์ |
| **Working MVP** | Software จริง แต่ feature น้อยมาก | 2-8 สัปดาห์ |

### MVP Scope Template
```
MVP นี้จะ validate: [สมมติฐาน Critical ข้อไหน]
Features ที่ต้องมี:
- [Feature 1] เพราะ [เหตุผล]
- [Feature 2] เพราะ [เหตุผล]

Features ที่ตัดออก:
- [Feature X] — เพิ่มได้ใน Sprint ถัดไป
- [Feature Y] — ยังไม่ต้องการตอน validate

เกณฑ์สำเร็จ (Success Metrics):
- [ตัวเลข 1]: [เป้าหมาย]
- [ตัวเลข 2]: [เป้าหมาย]

เวลาที่ใช้: [X สัปดาห์]
```

## Validation Results & Decision Framework

### หลังได้ข้อมูลจาก MVP แล้ว ต้องตัดสินใจ

```
PERSEVERE (ไปต่อ):
- สมมติฐาน Critical validate แล้ว
- ตัวเลขถึงเกณฑ์ที่กำหนด
- ลูกค้าพูดถึงปัญหาแบบเดิมซ้ำๆ

PIVOT (เปลี่ยนทิศ):
- สมมติฐาน Critical ผิด
- ลูกค้าไม่ยอมจ่าย แต่ยังสนใจ
- เจอ segment ลูกค้าอื่นที่ดีกว่า

STOP (หยุด):
- ไม่มี signal ใดๆ จากตลาด
- ลูกค้าไม่มีปัญหานี้จริง
```
